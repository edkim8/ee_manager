## Architectural Optimization

**Implement a schema validator for the `5p_Residents_Status` file to detect unit-assignment anomalies before committing tenancy state.**

Today's Hong, Jonathan (SB-2025) entry illustrates a category of silent data quality failure: a tenancy record with a past `lease_start_date` and Applicant status that flows through the solver without any validation gate. The solver correctly upserts the record and saves the application — but a tenancy with a start date 20 days in the past that is still Applicant is logically inconsistent and should trigger a warning.

**Proposed:** In the Applications processing phase, add a pre-commit check:

```ts
const today = new Date(uploadDateStr)
const startDate = new Date(tenancy.lease_start_date)
const daysInPast = (today.getTime() - startDate.getTime()) / (1000 * 60 * 60 * 24)

if (daysInPast > 7 && tenancy.status === 'Applicant') {
  console.warn(
    `[Solver] WARN: ${property} tenancy ${tenancy.id} (${tenancy.unit}) has ` +
    `Applicant status but lease_start_date is ${Math.round(daysInPast)} days in the past — ` +
    `expected Current or Canceled. Verify in Yardi.`
  )
}
```

A 7-day grace period accounts for weekend move-ins where Yardi may not convert status until Monday. Beyond 7 days, an Applicant with a past start date is almost certainly a data entry error. This surfaces the issue in the solver log and email immediately, rather than requiring forensic audit discovery. **Scope:** ~8 lines in the Applications processing phase. Zero DB impact — warning only.
