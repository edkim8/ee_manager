### Phase 1 — Yardi Delinquency Script Fix
**File:** `scripts/yardi-automation/phases/phase_c_analytics.py`

**Problem:** The `_run_delinquencies()` function had a `page.wait_for_timeout(2000)` static
wait after selecting `Report Type = "Financial Aged Receivable"`. Yardi fires an ASP.NET
`__doPostBack` that fully re-renders the iframe DOM. The 2000ms was racing against variable
network latency, letting the postback overwrite the `Summary Type` field back to "Property"
— causing the Excel export to be a Property-level report instead of Resident-level.

**Fix:**
- Replaced static wait with `page.expect_response(lambda r: "ARAnalytics_res.aspx" in r.url
  and r.request.method == "POST", timeout=20_000)` context manager — captures the exact
  server response before proceeding
- Re-acquires `#cmbGroupby_DropDownList` from the rebuilt DOM after the postback
- Adds immediate verification: if `Summary Type` label ≠ "Resident" → `RuntimeError` abort
- Adds `#btnDisplay_Button` click (confirmed selector from user) + 30s wait for table render
- **Column verification:** reads `table tr:first-child th/td` headers — aborts with
  `DO NOT DOWNLOAD` message if "Property Name" column is detected (wrong mode)
- Prints column headers to terminal for visual confirmation every run

**Selections confirmed correct:**
- Report Type: `Financial Aged Receivable`
- Summary Type: `Resident`
- Properties: `azres422 | azstoran | cacitvie | caoceabr | cawhioak` (pipe-separated, from `PHASE_C_PROPERTY_CODES`)
- Status: Current + Notice + Eviction (multi-select JS evaluation, unchanged)

7. Discrepancies
8. Status Auto-Fixes
9. Price Changes table (Unit | Property | Old | New | Change | %)
10. Availability Snapshots (available count + delta, avg rent + delta)

### Phase 5 — Data Quality Soft Checks (missing dates)

**Policy:** `move_in_date` (from 5p_Residents_Status) and `move_out_date` (from 5p_Notices /
5p_Residents_Status) are business-critical. Fields remain nullable in Supabase for the
transitional data-loading period, but the solver now warns when they're absent.

**Three check points in `useSolverEngine.ts`:**
1. New tenancy: `if (!tenancy.move_in_date)` → `console.warn` + `missing_move_in_date: true` in details
2. Notice: `if (!noticeMoveOutDate)` → `console.warn` (critical — overdue tracking cannot fire) + flag in details
3. Move-out (`→ Past`): checks both dates → `console.warn` per missing field + flags in details

**In `audit-export.get.ts`:**
- `missingDateEvents` = events where `details.missing_move_in_date === true` OR
  `details.missing_move_out_date === true`
- **DATA QUALITY section** in text payload: lists each affected unit with which date is missing,
  source file reminder (5p_Residents_Status / 5p_Notices), and resolution path

**No hard crashes. No aborts. Runs complete normally.**

## Key Architecture Reminders

- `solver_runs.summary` = JSON of `Record<propertyCode, PropertySummary>` — now includes
  `moveOutsProcessed` field (new this session)
- `solver_events.details` = `Record<string, any>` — now may include `missing_move_in_date`
  and/or `missing_move_out_date` boolean flags for data quality queries
- `event_type` values in use: `new_tenancy`, `move_out`, `notice_given`, `price_change`,
  `lease_renewal`, `silent_drop`, `discrepancy`, `status_auto_fix`, `application_saved`,
  `new_resident` (note: `lease_signed` is legacy/unused — was replaced by `new_tenancy`)
- Silent drop = Yardi removes tenancy from report entirely (no Past row). Current/Notice →
  inferred Past. Applicant/Future → inferred Canceled.
- `move_out` event = explicit Yardi `→ Past` transition (property manager closed the record)
- Supabase hosted only — no Docker. `npx supabase db push` for migrations.
- Dev server on port 3001. `ssr: false` on all authenticated routes (Vercel OOM fix).
